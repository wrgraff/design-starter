# Accessibility (A11y)

Accessibility is a baseline requirement, not an enhancement. The bar is **WCAG 2.1 Level AA**.
This repository treats accessibility as a hard quality gate: UI work is not complete until accessibility checks and tests pass.

This document is the operational checklist. The hard rules live in `AGENTS.md` → _Accessibility (Mandatory)_.

## Why It Is Treated as a Separate Layer

A11y issues compound: a single missing `aria-label` does not break anything, but ten of them across a codebase make the app unusable with assistive tech. Catching this requires:

1. **Defaults that are correct.** The UI primitives in `src/lib/components/ui/` enforce focus rings, ARIA semantics, and keyboard behavior. A feature that uses only primitives is mostly a11y-correct by construction.
2. **Compile-time checks.** `svelte-check` catches missing labels, redundant roles, unreachable elements, autofocus on non-interactive elements, etc.
3. **Automated runtime tests.** `@axe-core/playwright` against full pages and critical UI states.
4. **Manual checks.** Keyboard-only navigation, screen reader smoke test, contrast spot-checks on tricky tokens.

If any layer is skipped, a11y degrades over time.

## Baseline Requirements

### Semantic HTML

- `<button>` for things that do something on the current page.
- `<a href>` for things that go somewhere (route or external).
- `<input>` always inside or after a `<label for="...">`, or with an `aria-label`.
- Headings (`<h1>` … `<h6>`) follow a logical hierarchy. Do not skip levels for visual size — use tokens for size.
- Lists are `<ul>` / `<ol>` / `<dl>`, not divs.

Forbidden:

- `<div onclick>` — use `<button type="button">`.
- `<span role="button" tabindex="0" onclick>` — same; use `<button>`.
- `<a>` without `href` for in-app actions.
- Heading levels chosen by visual size rather than document hierarchy.

### Keyboard

- Every interactive element is reachable by Tab.
- Tab order matches visual reading order (top-to-bottom, left-to-right in LTR).
- `:focus-visible` styles are present and meet contrast.
- `Esc` closes modals/popovers/dropdowns. The `<Dialog>` primitive handles this.
- `Enter` and `Space` activate buttons. (Native `<button>` does this for free; custom elements rarely should be made.)
- No keyboard traps. After Tab past the last item, focus moves out (or wraps within a dialog if a focus trap is intentional).

Forbidden:

- `tabindex` greater than 0 (manual ordering).
- Removing focus styles (`outline: none`) without an equivalent replacement.

### Forms

- Every input has a `<label for="...">` or `aria-label` or `aria-labelledby`.
- Error messages are associated with their input via `aria-describedby` and announced (`aria-live="polite"` or `role="alert"`).
- Required fields are marked with `required` (not just a red asterisk).
- Validation does not rely on color alone — pair color with an icon and text.
- `autocomplete` attributes are set for known field types (`email`, `current-password`, `one-time-code`, etc.).

### Icons

- Icon-only buttons have `aria-label`.
- Decorative icons paired with text labels have `aria-hidden="true"`.
- Icons that convey information not in adjacent text are described via `aria-label` on the icon or `aria-describedby`.

### Color

- Text and meaningful icons meet 4.5:1 contrast against their background (3:1 for text ≥ 18pt regular / ≥ 14pt bold).
- UI components and graphical objects (icons, focus rings, form borders) meet 3:1.
- Information is never conveyed by color alone. Pair with text or icon.
- The token palette in `src/app.css` is calibrated to meet contrast in both light and dark themes — using tokens keeps you safe. Hardcoded colors usually do not.

### Images

- Every `<img>` has `alt`.
- Purely decorative images use `alt=""`.
- Complex images (charts, diagrams) have a longer description nearby or via `aria-describedby`.
- Background images (CSS) must not carry meaning — that meaning belongs in `<img>` or text.

### Motion

- Respect `prefers-reduced-motion`. Long, looping, or vestibular-triggering animations must be reduced or removed.
- The token system exposes `--transition-*` values that already account for this when used via the recommended pattern.

### Live Regions

- Toast notifications: container has `role="status"` (polite) or `role="alert"` (assertive).
- Loading states for async operations: announce start and completion via a polite live region or by managing focus.
- Search results / filtered lists: announce result count change via a polite live region.

### Dialogs and Modals

- Use the `<Dialog>` primitive. It handles focus trap, focus return, `aria-modal`, label, description, `Esc`, and backdrop click.
- Do not build modals from scratch.

## Testing

### Layer 1 — Compile-time

Run on every change. Lefthook enforces it.

```bash
pnpm check
```

`svelte-check` reports a11y warnings as errors in our config.

### Layer 2 — Route/state automated

`@axe-core/playwright` against pages and critical interactive states. Run before PR.

```bash
pnpm test:a11y
```

Failures block CI.

For strict project-wide coverage (all key routes and critical UI states), run:

```bash
pnpm test:a11y:full
```

This suite is expected to be updated whenever you add or change UI states.

### Layer 3 — Contrast for new tokens

Whenever you add a token to `src/app.css`, verify contrast for all `foreground` / `background` pairs you intend to use it in:

```bash
pnpm tokens:check-contrast
```

This script reads tokens from `app.css` and checks documented pairs against WCAG ratios. Add new pairs to `scripts/contrast-pairs.json` when introducing new combinations.

### Layer 4 — Manual

Before merging a feature that has its own UI:

- **Keyboard-only walkthrough.** Unplug the mouse. Tab through. Activate every control. Does it work? Is the focus visible at every step? Can you complete the primary flow?
- **Screen reader smoke test.** macOS: `Cmd+F5` for VoiceOver. Windows: NVDA. Walk the primary flow. Are buttons announced? Are form errors announced when they occur? Do live regions read out?
- **Zoom.** 200% zoom in the browser. Does layout still work? Does anything get cut off or unreachable?
- **Reduced motion.** macOS: System Settings → Accessibility → Display → Reduce motion. Reload. Do animations behave appropriately?
- **Dark theme.** Switch theme. Re-check contrast on anything that looked dense.

## Required Workflow for UI Changes

If a task changes any user-facing UI, it must include all of the following:

1. Add or update `@a11y` / `@a11y-full` tests for the affected route/component/state.
2. Run `pnpm check`.
3. Run `pnpm tokens:check-contrast` when color/token usage changes.
4. Run `pnpm test:a11y:full`.

If any of these are skipped, the change is incomplete.

## Common Mistakes to Avoid

| Mistake                                    | Fix                                               |
| ------------------------------------------ | ------------------------------------------------- |
| `<div onclick={...}>`                      | `<button type="button" onclick={...}>`            |
| `<a onclick={...}>` (no href)              | `<button>`                                        |
| Tooltip-only labels on icon buttons        | `aria-label` on the button                        |
| Red-bordered invalid input, no text        | Add error text + `aria-describedby`               |
| `placeholder` instead of `<label>`         | Add `<label>`; keep placeholder for examples only |
| `<h2>` chosen because it looked right size | Pick by document hierarchy; size via tokens       |
| Custom dropdown built from scratch         | Use the `<Select>` / `<Dialog>` primitive         |
| Modal that focuses nothing on open         | `<Dialog>` handles this — use it                  |
| Loading spinner with no announcement       | Add `aria-live` region or focus management        |
| Color-only "you have 3 new" indicator      | Add text or `aria-label`                          |
| Decorative background image carrying info  | Move the info into the DOM as text                |

## When You Are Unsure

- Read `docs/COMPONENTS.md` for the a11y contract of each primitive.
- Search axe-core's rule catalog: <https://dequeuniversity.com/rules/axe/>.
- Ask the maintainer rather than guess. A11y is one area where guessing usually goes wrong.
